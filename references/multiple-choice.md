# 객관식 프롬프팅과 객관식 플래닝

객관식은 참가자의 답을 제한하기 위한 장치가 아니라, 빠르게 비교하고 합의하기 위한 수렴 장치입니다. 사용자가 이미 명확히 답한 내용은 객관식으로 다시 묻지 않습니다. 답이 추상적이거나 여러 방향이 섞였을 때만 사용합니다.

나쁜 객관식은 `추천안 1개 + 불필요한 들러리 2개`입니다. 좋은 객관식은 모두 선택해도 말이 되는 2-3개의 실행 가능한 대안입니다. 각 선택지는 같은 문제를 다른 관점으로 해결해야 하며, 사용자의 우선순위에 따라 실제로 선택이 갈릴 수 있어야 합니다.

선택지를 만들기 전에 사용자의 핵심 자동화 의도를 먼저 판별합니다. 사용자가 "AI가 산출물을 만들어주길" 기대한다면 모든 선택지는 자동 생성형이어야 합니다. 검수표, 체크리스트, 입력 보조는 독립 선택지가 아니라 자동 생성 결과를 안전하게 쓰기 위한 보강 요소로 둡니다.

## 두 가지 사용법

### 객관식 프롬프팅

하나의 빈칸을 빠르게 채울 때 씁니다.

- 문제 유형 선택
- 데모 산출물 형식 선택
- 사람 검토 방식 선택
- 제외 범위 선택
- 3시간 밖으로 보일 때 줄일 축 선택

형식:

```text
제가 보기엔 `[추천 선택지]`가 가장 맞아 보입니다.

A. [선택지 이름]: [한 줄 설명]
B. [선택지 이름]: [한 줄 설명]
C. [선택지 이름]: [한 줄 설명]

추천: [A/B/C]
이유: [현장 문제, 샘플 데이터, 3시간 제약을 근거로 한 한 문장]

A/B/C 중 고르거나, 섞고 싶은 부분을 말해주세요.
```

### 객관식 플래닝

MVP 방향 자체를 2-3개 완성안으로 비교할 때 씁니다.

- 여러 MVP 후보가 나왔을 때
- 사회혁신가가 "너무 단순하다"고 느낄 수 있을 때
- 시스템 구축 욕구를 3시간 데모로 줄여야 할 때
- 3시간 안/밖 판단이 `축소 필요`일 때

형식:

```text
3시간 MVP 후보를 세 가지로 자르면 이렇게 됩니다.

A. [안 이름]
- 입력: [샘플 입력 업무 묶음]
- 흐름: [2-4개 업무 단계]
- 산출물: [검토 가능한 최종 산출물]
- 장점: [현장 효용]
- 위험: [시간/데이터/검수 위험]
- 빠지는 것: [오늘 하지 않을 것]

B. [안 이름]
- 입력: [샘플 입력 업무 묶음]
- 흐름: [2-4개 업무 단계]
- 산출물: [검토 가능한 최종 산출물]
- 장점: [현장 효용]
- 위험: [시간/데이터/검수 위험]
- 빠지는 것: [오늘 하지 않을 것]

C. [안 이름]
- 입력: [샘플 입력 업무 묶음]
- 흐름: [2-4개 업무 단계]
- 산출물: [검토 가능한 최종 산출물]
- 장점: [현장 효용]
- 위험: [시간/데이터/검수 위험]
- 빠지는 것: [오늘 하지 않을 것]

추천: [A/B/C]
이유: [왜 이 안이 3시간 안에 가장 완결적인지]

어느 안으로 갈까요? A/B/C로 답해도 되고, 섞어서 조정해도 됩니다.
```

## 운영 규칙

- 선택지는 2-3개를 기본으로 하고, 4개를 넘기지 않습니다.
- 항상 추천안을 하나 제시합니다.
- 추천 이유는 한 문장으로 씁니다.
- 각 선택지는 서로 겹치지 않게 만듭니다.
- 명백히 약하거나 불필요한 선택지를 넣지 않습니다.
- 이미 제외하기로 한 기능, 3시간 밖의 시스템 구축, 단순 축소판은 선택지로 넣지 않습니다.
- 모든 선택지는 사용자의 핵심 자동화 의도를 같은 수준으로 충족해야 합니다.
- 사용자가 반복 제작물을 줄이고 싶어 하면 모든 선택지는 제작물 자동 생성형이어야 합니다.
- 사용자가 추천이나 매칭을 원하면 모든 선택지는 추천/매칭 결과를 포함해야 합니다.
- 사용자가 검수나 입력 오류 자체를 핵심 문제로 말한 경우에만 검수/입력 보조를 독립 선택지로 둡니다.
- 각 선택지는 서로 다른 성공 기준을 가져야 합니다. 성공 기준은 이 프로젝트의 입력, 결과물, 사람 검토 지점을 기준으로 작성하고 일반적인 지표 목록을 그대로 제시하지 않습니다.
- 선택지마다 `장점`, `위험`, `빠지는 것`을 드러냅니다.
- 번호만 답해도 다음 단계로 진행합니다.
- 사용자가 선택지를 섞으면 새 합의안을 1-2문장으로 요약하고 진행합니다.
- 객관식이 길어지면 먼저 추천안만 제시하고, 사용자가 원할 때 다른 안을 펼칩니다.

## 선택지 품질 점검

객관식을 제시하기 전에 아래를 확인합니다.

- 사용자가 당연히 고를 선택지가 하나뿐이면 다시 만듭니다.
- 세 선택지가 모두 같은 문제를 해결하는가?
- 세 선택지가 모두 사용자의 핵심 자동화 기대를 충족하는가?
- 세 선택지가 모두 3시간 안에 데모 가능한가?
- 각 선택지가 서로 다른 장점과 위험을 갖는가?
- 선택하지 않은 안도 후속 후보나 다음 단계로 남길 만큼 의미가 있는가?
- 선택지가 기능 목록이 아니라 완성된 업무 흐름으로 표현됐는가?
- 검수/체크리스트/입력 보조가 자동 생성형 선택지를 밀어내고 있지 않은가?

## 단계별 선택지 템플릿

### 문제 유형

```text
제가 보기엔 A가 가장 맞아 보입니다.

A. [프로젝트 기반 안 A]: [이 프로젝트의 입력] -> [처리 흐름] -> [사람 검토 산출물]
B. [프로젝트 기반 안 B]: [같은 문제의 다른 실행 흐름] -> [검토 가능한 산출물]
C. [프로젝트 기반 안 C]: [세 번째 실행 가능한 흐름] -> [현장에서 바로 볼 결과]

추천: A
이유: [이 프로젝트의 자료 상태, 3시간 제약, 사람 검토 지점을 근거로 한 한 문장]

A/B/C 중 고르거나, 섞고 싶은 부분을 말해주세요.
```

### MVP 방향

```text
이 프로젝트의 3시간 MVP 방향은 세 가지로 볼 수 있습니다.

A. [프로젝트 기반 방향 A]
- 입력: [이 프로젝트의 샘플 입력 업무 묶음]
- 흐름: [이 프로젝트의 2-4개 업무 단계]
- 산출물: [사람이 검토 가능한 최종 산출물]
- 장점: [이 프로젝트에서 바로 줄어드는 일]
- 위험: [이 프로젝트의 시간/자료/검수 위험]
- 빠지는 것: [오늘 하지 않을 것]

B. [프로젝트 기반 방향 B]
- 입력: [같은 문제를 다른 단위로 자른 샘플 입력]
- 흐름: [다른 2-4개 업무 단계]
- 산출물: [다른 검토 가능한 최종 산출물]
- 장점: [이 프로젝트에서 A와 달리 좋아지는 점]
- 위험: [이 프로젝트에서 A보다 어려운 점]
- 빠지는 것: [오늘 하지 않을 것]

C. [프로젝트 기반 방향 C]
- 입력: [세 번째로 실행 가능한 샘플 입력]
- 흐름: [세 번째 2-4개 업무 단계]
- 산출물: [세 번째 검토 가능한 최종 산출물]
- 장점: [이 프로젝트에서 C가 주는 현장 효용]
- 위험: [이 프로젝트에서 C의 검수/시간 위험]
- 빠지는 것: [오늘 하지 않을 것]

추천: B
이유: [왜 이 안이 이 프로젝트에서 3시간 안에 가장 완결적인지]

어느 안으로 갈까요?
```

### 3시간 밖일 때 축소 선택

```text
지금 범위는 3시간을 넘을 가능성이 큽니다. 줄이는 방법은 세 가지입니다.

A. 입력 줄이기: 이 프로젝트의 전체 자료 대신 발표 가능한 최소 입력 묶음으로 제한
B. 단계 줄이기: 이 프로젝트의 전체 흐름 대신 사람이 검토할 핵심 산출물까지만 생성
C. 연동 빼기: 이 프로젝트에서 필요한 외부 연동을 다음 단계로 두고 수동 입력과 복사용 결과로 시연

추천: C
이유: [무엇을 빼도 이 프로젝트의 핵심 판단과 산출물은 보여줄 수 있는지]

A/B/C 중 무엇을 줄일까요?
```
