# 객관식 프롬프팅과 객관식 플래닝

객관식은 참가자의 답을 제한하기 위한 장치가 아니라, 빠르게 비교하고 합의하기 위한 수렴 장치입니다. 사용자가 이미 명확히 답한 내용은 객관식으로 다시 묻지 않습니다. 답이 추상적이거나 여러 방향이 섞였을 때만 사용합니다.

## 두 가지 사용법

### 객관식 프롬프팅

하나의 빈칸을 빠르게 채울 때 씁니다.

- 문제 유형 선택
- 데모 산출물 형식 선택
- 사람 검토 방식 선택
- 제외 범위 선택
- 3시간 밖으로 보일 때 줄일 축 선택

형식:

```text
제가 보기엔 `[추천 선택지]`가 가장 맞아 보입니다.

A. [선택지 이름]: [한 줄 설명]
B. [선택지 이름]: [한 줄 설명]
C. [선택지 이름]: [한 줄 설명]

추천: [A/B/C]
이유: [현장 문제, 샘플 데이터, 3시간 제약을 근거로 한 한 문장]

A/B/C 중 고르거나, 섞고 싶은 부분을 말해주세요.
```

### 객관식 플래닝

MVP 방향 자체를 2-3개 완성안으로 비교할 때 씁니다.

- 여러 MVP 후보가 나왔을 때
- 사회혁신가가 "너무 단순하다"고 느낄 수 있을 때
- 시스템 구축 욕구를 3시간 데모로 줄여야 할 때
- 3시간 안/밖 판단이 `축소 필요`일 때

형식:

```text
3시간 MVP 후보를 세 가지로 자르면 이렇게 됩니다.

A. [안 이름]
- 입력: [샘플 입력 업무 묶음]
- 흐름: [2-4개 업무 단계]
- 산출물: [검토 가능한 최종 산출물]
- 장점: [현장 효용]
- 위험: [시간/데이터/검수 위험]
- 빠지는 것: [오늘 하지 않을 것]

B. [안 이름]
- 입력: [샘플 입력 업무 묶음]
- 흐름: [2-4개 업무 단계]
- 산출물: [검토 가능한 최종 산출물]
- 장점: [현장 효용]
- 위험: [시간/데이터/검수 위험]
- 빠지는 것: [오늘 하지 않을 것]

C. [안 이름]
- 입력: [샘플 입력 업무 묶음]
- 흐름: [2-4개 업무 단계]
- 산출물: [검토 가능한 최종 산출물]
- 장점: [현장 효용]
- 위험: [시간/데이터/검수 위험]
- 빠지는 것: [오늘 하지 않을 것]

추천: [A/B/C]
이유: [왜 이 안이 3시간 안에 가장 완결적인지]

어느 안으로 갈까요? A/B/C로 답해도 되고, 섞어서 조정해도 됩니다.
```

## 운영 규칙

- 선택지는 2-3개를 기본으로 하고, 4개를 넘기지 않습니다.
- 항상 추천안을 하나 제시합니다.
- 추천 이유는 한 문장으로 씁니다.
- 각 선택지는 서로 겹치지 않게 만듭니다.
- 선택지마다 `장점`, `위험`, `빠지는 것`을 드러냅니다.
- 번호만 답해도 다음 단계로 진행합니다.
- 사용자가 선택지를 섞으면 새 합의안을 1-2문장으로 요약하고 진행합니다.
- 객관식이 길어지면 먼저 추천안만 제시하고, 사용자가 원할 때 다른 안을 펼칩니다.

## 단계별 선택지 예시

### 문제 유형

```text
제가 보기엔 A가 가장 맞아 보입니다.

A. 검토 보조형: 입력 묶음 -> 요약/분류 -> 근거 -> 담당자 검토표
B. 전달 산출물형: 입력 묶음 -> 초안 -> 공유 메시지/메일/안내문
C. 운영 정리형: 여러 건 입력 -> 우선순위표 -> 다음 행동 목록

추천: A
이유: 최종 판단을 자동화하지 않고도 담당자가 바로 검토할 근거를 만들 수 있기 때문입니다.

A/B/C 중 고르거나, 섞고 싶은 부분을 말해주세요.
```

### MVP 깊이

```text
3시간 MVP 깊이는 세 가지로 볼 수 있습니다.

A. 안전한 완결형
- 입력: 샘플 3건
- 흐름: 요약 -> 분류 -> 근거 표시 -> 검토 체크리스트
- 산출물: 담당자 검토표
- 장점: 완성 가능성이 높습니다.
- 위험: 발표에서 시스템처럼 보이지 않을 수 있습니다.
- 빠지는 것: 자동 발송, DB 저장

B. 깊이 있는 검토형
- 입력: 샘플 3-5건
- 흐름: 요약 -> 분류 -> 근거 표시 -> 보류 사유 -> 회의 브리핑
- 산출물: 회의용 표와 브리핑 문단
- 장점: 사회혁신가가 실무 효용을 더 크게 느낄 수 있습니다.
- 위험: 출력 품질 검수 시간이 더 필요합니다.
- 빠지는 것: 운영 화면, 권한 관리

C. 시스템처럼 보이는 데모형
- 입력: 화면 입력 1개
- 흐름: 입력 -> 결과 화면
- 산출물: 간단한 웹 화면
- 장점: 발표에서 보기 쉽습니다.
- 위험: 실제 업무 완결성보다 UI 구현에 시간이 쓰일 수 있습니다.
- 빠지는 것: 깊은 검토 기준, 예외 처리

추천: B
이유: 3시간 안에서 단순 초안을 넘어 실제 회의나 검토에 쓸 수 있는 결과를 만들 수 있기 때문입니다.

어느 안으로 갈까요?
```

### 3시간 밖일 때 축소 선택

```text
지금 범위는 3시간을 넘을 가능성이 큽니다. 줄이는 방법은 세 가지입니다.

A. 입력 줄이기: 전체 데이터를 샘플 3-5건으로 제한
B. 단계 줄이기: 자동 실행 대신 검토 가능한 표와 메시지만 생성
C. 연동 빼기: API/DB/발송 없이 수동 업로드와 복사용 결과로 시연

추천: C
이유: 연동을 빼도 현장 업무의 핵심 판단과 산출물은 보여줄 수 있기 때문입니다.

A/B/C 중 무엇을 줄일까요?
```
