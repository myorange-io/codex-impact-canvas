# 캔버스 진행 흐름

이 흐름은 한국어로 진행합니다. 한 번에 하나의 질문만 묻고, 답변을 받은 뒤 1-2문장으로 요약합니다. 이 스킬은 신청서 내용을 다시 묻는 인터뷰가 아니라, 참여자가 준비해 온 현장 문제를 3시간 MVP 1개와 AI Agent Workflow 자산으로 압축하는 워크숍 코치입니다.

## 목차

- 질문 도구 사용
- 6단계 진행 흐름
- 체크리스트로 쓰는 기존 앵커
- 수렴 규칙
- 산출물 안내
- 구현 핸드오프
- 발표자료 핸드오프

## 질문 도구 사용

이 문서의 `질문`, `부족할 때만 묻는 질문`, `수렴 규칙` 문구는 `request_user_input` 도구가 활성화되어 있을 때는 도구 호출의 재료로 쓰고, 도구가 비활성화되어 있을 때는 같은 순서의 일반 채팅 질문으로 씁니다.

- 사용자에게 새 정보를 받아야 할 때 현재 모드에서 `request_user_input`이 활성화되어 있으면 도구를 우선 사용합니다.
- `request_user_input`이 비활성화된 Default mode에서는 Plan mode 전환이 필요하다고 짧게 안내하고, 바로 진행할 때는 일반 채팅 질문으로 이어갑니다.
- 도구를 사용할 때는 한 번의 호출에 질문 1개만 넣고, 답변을 받은 뒤 요약한 다음 다음 호출 여부를 판단합니다.
- 이미 확인된 내용은 다시 묻지 않습니다. 빈칸이 있을 때만 질문합니다.
- 자유 서술 답변이 필요한 단계에서는 도구가 활성화되어 있으면 선택지를 답변 방식으로 구성합니다. 예: `짧게 설명`, `자료를 먼저 붙여넣기`, `가상 샘플로 시작`. 도구가 자유 입력 옵션을 자동으로 붙이므로 `직접 입력` 옵션을 별도로 만들지 않습니다.
- 유형 선택, 결과물 선택, MVP 압축처럼 선택이 필요한 단계에서는 모든 선택지가 실제로 3시간 MVP가 될 수 있어야 합니다. 명백한 정답 하나와 들러리 선택지를 만들지 않습니다.
- 답변이 추상적이거나 여러 방향이 섞이면 `references/multiple-choice.md`를 사용해 객관식 프롬프팅 또는 객관식 플래닝으로 수렴합니다.
- 선택지 설명은 한 줄로 쓰고, 그 선택을 고르면 무엇이 확정되는지 밝힙니다.

도구가 활성화되어 있을 때의 기본 호출 형태:

```text
request_user_input
question: "[아래 단계의 질문 문구를 상황에 맞게 한 문장으로 압축]"
multiSelect: false
options:
  - label: "[추천 또는 가장 자연스러운 답변 방식]"
    description: "[이 선택으로 확정되는 것]"
  - label: "[다른 실행 가능한 선택]"
    description: "[이 선택으로 확정되는 것]"
```

## 6단계 진행 흐름

### 1. 기본 정보 확인

먼저 팀 시작 정보를 받습니다.

- 사회혁신가 이름
- 개발자 이름
- 팀 이름 또는 프로젝트 이름. 이미 정해져 있을 때만 받습니다.

프로젝트 이름이 없다면 문제 설명에서 짧은 임시 제목을 만들고, 임시 제목이라고 표시합니다.

질문:

```text
사회혁신가 이름은 무엇인가요?
```

답변을 받은 뒤 묻는 질문:

```text
개발자 이름은 무엇인가요?
```

답변을 받은 뒤 묻는 질문:

```text
정해진 팀 또는 프로젝트 이름이 있나요?
```

완료 기준:

- 사회혁신가와 개발자 이름이 기록됩니다.
- 팀명 또는 프로젝트명이 있으면 기록됩니다.
- 프로젝트명이 없으면 임시 제목을 만들 수 있습니다.

### 2. 업무 문제 확인

샘플 입력이나 기대 산출물을 바로 묻지 않습니다. 먼저 실제 업무 장면을 확인합니다. 기본 질문은 2-3개로 나눠 묻습니다. 한 답변에 반복 업무, 막히는 사람, 막히는 순간이 모두 들어 있으면 남은 후속 질문은 생략합니다.

첫 질문:

```text
지금 반복해서 처리하느라 가장 불편한 업무 1개는 무엇인가요?
```

답변에 어려움을 겪는 사람이 빠졌을 때만 묻는 질문:

```text
이 업무에서 가장 막히는 사람은 누구인가요? 직접 처리하는 사람, 검토자, 최종 수혜자 중 누가 가장 불편한지 알려주세요.
```

답변에 막히는 순간이 빠졌을 때만 묻는 질문:

```text
그 사람이 실제로 막히는 순간은 언제인가요? 예를 들어 신청서가 몰릴 때, 상담 기록을 다시 읽을 때, 결과를 안내문으로 바꿀 때처럼 한 장면으로 말해 주세요.
```

이미 세 요소를 한 번에 물어야 하는 상황에서만 쓰는 압축 질문:

```text
지금 반복해서 처리하느라 불편한 업무는 무엇이고, 누가 어떤 순간에 가장 막히나요?
```

완료 기준:

- 반복 처리 업무가 드러납니다.
- 불편을 겪는 사람이 직접 사용자, 검토자, 최종 수혜자 중 누구인지 구분됩니다.
- 문제가 발생하는 순간이 한 문장으로 정리됩니다.

추가로 부족할 때만 묻는 질문:

- "가장 자주 반복되는 업무 1개만 고르면 무엇인가요?"
- "이 문제 때문에 가장 시간을 쓰거나 곤란해지는 사람은 누구인가요?"
- "문제가 실제로 터지는 순간을 하나만 말하면 언제인가요?"

### 3. 자료 확인

업무 문제를 들은 뒤 실제 자료 상태를 확인합니다. 자료가 없으면 가상 샘플을 만들 수 있습니다.

질문:

```text
오늘 써볼 실제 자료가 있나요?

오렌지필터는 개인정보와 민감정보를 익명화해 워크숍용 샘플을 만들 때 쓰는 Chrome 확장 프로그램입니다. 자료가 아직 익명화되지 않았다면 먼저 오렌지필터로 처리한 뒤 익명화된 샘플만 사용합니다.
설치 링크: https://chromewebstore.google.com/detail/%EC%98%A4%EB%A0%8C%EC%A7%80%ED%95%84%ED%84%B0orange-filter-ai-%ED%94%84%EB%9D%BC%EC%9D%B4/gnkcbdgbajeboglkplllicdghlcghjai

자료가 없다면 오늘은 가상 샘플로 시작해도 됩니다.
```

완료 기준:

- 실제 자료 유무가 정해집니다.
- 자료 형태가 파일, 표, 문서, 질문 묶음, 화면, 텍스트, 없음 중 하나로 기록됩니다.
- 개인정보 처리 방식이 익명화, 마스킹, 삭제, 더미 생성, 화면 시연 중 하나로 정해집니다.
- 익명화되지 않은 샘플은 원본 그대로 사용하지 않고 오렌지필터 설치 후 익명화하도록 안내합니다.
- 자료가 없으면 스킬이 만들 가상 샘플의 조건이 정해집니다.

자료 유효성 확인 기준:

- 3시간 안에 열어보고 처리할 수 있는 크기입니다.
- 발표장에서 보여줘도 되는 수준으로 개인정보와 내부 원문이 제거되어 있습니다.
- 입력 -> 처리 -> 판단 -> 산출물 흐름을 보여줄 최소 정보가 있습니다.
- 원본을 공개할 수 없으면 `가상 샘플` 또는 `익명화 샘플`로 표시합니다.

오렌지필터 안내 문구:

```text
이 샘플은 아직 익명화가 필요합니다. 오렌지필터는 개인정보와 민감정보를 익명화해 워크숍용 샘플을 만들 때 쓰는 Chrome 확장 프로그램입니다. 원본을 그대로 쓰지 말고 먼저 오렌지필터를 설치해 익명화한 뒤, 익명화된 샘플만 워크숍에 사용해 주세요.
설치 링크: https://chromewebstore.google.com/detail/%EC%98%A4%EB%A0%8C%EC%A7%80%ED%95%84%ED%84%B0orange-filter-ai-%ED%94%84%EB%9D%BC%EC%9D%B4/gnkcbdgbajeboglkplllicdghlcghjai
```

### 4. 결과물 선택

`references/problem-archetypes.md`로 문제 유형을 가늠하고, `references/output-types.md`로 가능한 결과물 후보를 제안합니다. 이 단계에서 스킬이 후보를 먼저 만듭니다.

질문:

```text
이 문제는 `[문제 유형]`에 가깝습니다. 오늘 결과물은 아래 후보 중 무엇으로 보여주는 것이 가장 자연스러울까요?
```

완료 기준:

- 문제 유형이 문서/콘텐츠, 데이터/대시보드, 문의응대/매뉴얼, 분류/추천/리서치, 웹앱/시스템, MVP 미정형 중 하나로 기록됩니다.
- 결과물 형태가 자동화 도구, 웹앱, 온보딩 페이지, admin 화면, 대시보드, 문서/메시지 초안, 기타 중 하나로 기록됩니다.
- 결과물 후보는 모두 실행 가능해야 하며, 하나만 진짜 정답처럼 보이면 다시 구성합니다.

### 5. 3시간 MVP 좁히기

결과물 형태가 정해진 뒤 `references/mvp-scope.md`를 사용해 좁은 문제 1개, 입력 묶음 1개, 검토 가능한 산출물 1개, 사람 검토 1개로 완결형 업무 흐름을 자릅니다.

질문:

```text
3시간 안에 보여줄 핵심 흐름 1개만 남기면, 어떤 입력을 넣어 어떤 결과를 보여주고 사람이 어디를 검토하면 될까요?
```

성공/실패 기준을 확인할 때는 예시를 먼저 붙입니다. 실제 분야에 맞춰 바꿔 말하되, 샘플 입력으로 판정 가능한 문장이어야 합니다.

```text
예를 들어 신청서 검토라면 "샘플 신청서 1건에서 누락 항목 2개 이상과 근거가 나오면 성공, 필수 누락을 놓치거나 개인정보가 결과에 남으면 실패"처럼 정할 수 있습니다. 이 MVP에서는 샘플 입력 1개를 넣었을 때 어떤 결과가 나오면 성공이고, 어떤 결과가 나오면 실패인가요?
```

완료 기준:

- 핵심 기능 1개가 있습니다.
- 해결할 좁은 문제와 포함할 업무 단계가 있습니다.
- 데모 입력과 데모 산출물이 정해집니다.
- 오늘 할 것과 하지 않을 것이 분리됩니다.
- AI 역할과 사람 검토 지점이 분리됩니다.
- 성공 기준과 실패 신호가 샘플 결과로 판단 가능합니다.
- 3시간 안/밖 판단이 `안`, `축소 필요`, `밖` 중 하나로 기록됩니다.
- 핵심 기능 1개가 끝났을 때 이어서 할 작업 후보 2개가 있습니다.
- 각 후보에는 시작 조건, 구현 내용, 데모 산출물, 예상 추가 시간, 검수 기준이 있습니다.

부족할 때만 묻는 질문:

- "AI가 맡을 일과 사람이 반드시 판단할 일을 나누면 어떻게 되나요?"
- "오늘 하지 않을 일 1-3개를 고르면 무엇인가요?"
- "예를 들어 상담 기록 요약이라면 핵심 위험 신호와 확인 질문이 나오면 성공, 위험 신호를 놓치거나 민감한 원문 표현이 남으면 실패입니다. 이 경우에는 무엇을 성공/실패로 볼까요?"
- "이 범위는 3시간 안, 축소 필요, 밖 중 어디에 가깝고 그 이유는 무엇인가요?"
- "핵심 기능 1개가 빨리 끝나면 이어서 만들 후보 2개는 무엇이고, 각각 언제 시작할 수 있나요?"

### 6. 다음 단계 안내

문제정의가 끝나면 `PLAN.md`와 `workshop.json` 초안을 만듭니다. `PLAN.md` 작성이 완료되면 파일을 다시 읽고 그 내용을 채팅창에 보여줍니다. 이후 스킬이 직접 구현을 시작하지 않고, 사용자에게 `PLAN.md` 기준으로 구현을 진행하라고 넘깁니다. 산출물 범위는 질문하지 않고 워크숍 표준 흐름을 안내합니다.

안내 문구:

```text
문제정의는 `PLAN.md`와 `workshop.json` 초안으로 저장했습니다. 아래에 `PLAN.md` 내용을 그대로 보여드리겠습니다.

[PLAN.md 전체 내용]

이제 구현을 시작하면 됩니다. 구현은 `PLAN.md`의 3시간 구현 계획을 기준으로 진행하고, 새 세션에서 구현하더라도 구현자는 `MEMORY.md`를 append-only로 갱신해야 합니다. 구현이 완료되면 같은 폴더에서 다시 이 스킬에 발표자료를 만들어달라고 요청하면 됩니다. 그때 표준 흐름에 따라 `WORKFLOW_ANALYSIS.md`, `CASE_STUDY.md`, 최종 `workshop.json`, 5장 발표자료 제작을 이어갑니다.
```

완료 기준:

- `PLAN.md` 초안의 용도가 팀 합의와 3시간 실행 계획임을 안내했습니다.
- `workshop.json`의 용도가 제출/아카이빙/산출물 재생성 기준 데이터임을 안내했습니다.
- 작성된 `PLAN.md` 전체 내용을 채팅창에 보여줬습니다.
- `PLAN.md` 작성 뒤 스킬이 직접 구현하지 않고 사용자에게 구현을 넘긴다는 점을 안내했습니다.
- `PLAN.md`에 새 세션 구현자가 `MEMORY.md`를 append-only로 갱신해야 한다는 구현자 지침을 포함했습니다.
- 이제 구현을 시작하면 되고, 구현 완료 후 발표자료를 만들어달라고 요청하면 된다는 다음 행동을 명확히 안내했습니다.
- 구현 완료 후에는 산출물 범위를 다시 묻지 않고 `MEMORY.md`, `WORKFLOW_ANALYSIS.md`, `CASE_STUDY.md`, 최종 `workshop.json`, 발표자료 제작을 표준 흐름으로 진행한다고 안내했습니다.

## 체크리스트로 쓰는 기존 앵커

아래 항목은 순서대로 반복 질문하지 않습니다. 사용자가 준비해 온 설명에서 이미 확인된 내용은 건너뛰고, 빈칸만 짧게 묻습니다.

- 실제 업무 문제: 누가, 언제, 무엇 때문에 막히는가?
- 어려움을 겪는 사람: 직접 사용자, 검토자, 최종 수혜자가 구분되는가?
- 현재 업무 흐름: 입력 -> 처리 -> 판단 -> 산출물이 보이는가?
- 자료 상태: 실제 자료, 익명화 샘플, 가상 샘플 중 무엇을 쓸 것인가?
- 결과물 형태: 발표장에서 무엇을 눈으로 보여줄 것인가?
- 기대 변화: 시간 절감, 오류 감소, 누락 방지, 응답 속도 개선, 일관성 개선 중 관찰 가능한 변화가 있는가?
- 3시간 MVP: 입력 1개, 출력 1개, 사용자 행동 1개, 사람 검토 1개로 줄었는가?
- 성공 판단 기준: 샘플 결과를 보고 성공/실패를 판단할 수 있는가?
- 제출/아카이빙: 최종 결과물과 기록이 워크플로 라이브러리로 재사용 가능한가?

## 수렴 규칙

팀이 여러 아이디어를 나열하면 하나의 완결형 업무 흐름을 고르게 합니다.

```text
3시간 MVP에서는 하나만 남기겠습니다. 오늘은 `[입력]`을 넣어 `[결과물]`을 만들고 `[사람 검토]`에서 확인하는 흐름으로 두고, `[나머지]`는 제외 범위나 다음 단계로 기록해도 될까요?
```

시스템급 요청은 오늘 만들 화면, 문서, 표, 프롬프트, 워크플로 중 하나로 줄입니다.

```text
이건 완성된 시스템으로는 3시간을 넘습니다. 오늘은 `[사용자 행동 1개]`와 `[결과 화면/문서 1개]`가 보이는 데모로 자르고, 연동과 자동 실행은 다음 단계로 빼겠습니다.
```

답변이 계속 추상적이면 반복 업무 하나부터 고르게 합니다.

```text
아직 기능이 정해지지 않았다면, 가장 자주 반복되는 업무 하나만 먼저 고르겠습니다. 오늘 사람이 덜 해도 되는 일 1개는 무엇인가요?
```

자료가 없으면 가상 샘플을 제안합니다.

```text
실제 자료가 없어도 진행할 수 있습니다. 오늘은 `[업무 상황]`을 반영한 가상 샘플 1개로 입력과 결과물을 고정해도 될까요?
```

## 산출물 안내

산출물 범위는 사용자에게 선택 질문으로 묻지 않습니다. 워크숍 표준 흐름을 기본으로 적용하되, 단계별 역할은 아래처럼 구분합니다.

- 문제정의 후: `PLAN.md`, `workshop.json` 초안
- 사용자가 구현 중: 구현자가 `MEMORY.md` append, `workshop.json` 업데이트 대상 결정
- 사용자가 구현 완료 후: `MEMORY.md`, `WORKFLOW_ANALYSIS.md`, `CASE_STUDY.md`, 최종 `workshop.json`
- 발표자료 제작 전: `workshop.json` 기준 입력 폴더, 선택적 `presentation-assets/result_screenshot.png`

제출/아카이빙 기준:

- `PLAN.md`에는 팀 합의, 자료 상태, 결과물 형태, 3시간 MVP, 제외 범위가 있습니다.
- `PLAN.md`에는 핵심 기능이 끝났을 때 이어서 할 작업 후보 2개가 있습니다.
- `workshop.json`에는 질문 답변과 구현 결과가 표준 필드로 들어 있습니다.
- `MEMORY.md`에는 구현 중 결정과 변경이 누적되어 있습니다.
- `WORKFLOW_ANALYSIS.md`에는 재사용 가능한 AI Agent Workflow 필드가 채워져 있습니다.
- `CASE_STUDY.md`에는 공개 가능한 수준의 익명화 사례만 들어 있습니다.
- 발표자료 제작 전에는 최종 `workshop.json`을 기준 파일로 사용합니다. `input.json`이 남아 있으면 최종 데이터와 충돌하지 않는지 확인합니다.
- 결과물 캡처가 필요한 팀은 공개 가능한 이미지만 `presentation-assets/result_screenshot.png`에 둡니다.

## 구현 핸드오프

`PLAN.md`와 `workshop.json` 초안이 만들어지면 이 스킬은 바로 구현을 시작하지 않습니다. 먼저 생성된 `PLAN.md` 전체 내용을 채팅창에 보여준 뒤, 사용자에게 아래를 짧게 넘기고 멈춥니다.

- `PLAN.md`의 3시간 구현 계획을 기준으로 구현을 진행합니다.
- 새 세션에서 구현하더라도 구현 중 결정, 변경, 막힌 점, 검증 결과를 `MEMORY.md`에 append-only로 기록합니다.
- 이제 구현을 시작하면 됩니다.
- 구현이 끝나면 같은 폴더에서 다시 스킬을 호출해 발표자료를 만들어달라고 요청합니다.
- 발표자료 제작 요청을 받으면 `MEMORY.md`, `WORKFLOW_ANALYSIS.md`, `CASE_STUDY.md`, 최종 `workshop.json`, 발표자료 제작으로 이어갑니다.

## 발표자료 핸드오프

워크숍 기록과 사례 정리가 끝나면 `references/presentation-handoff.md`와 `references/presentation-build.md`를 확인합니다. 이 단계의 목표는 참가팀 폴더를 같은 `codex-impact-canvas` 스킬 안에서 5장 Google Slides로 만들 수 있는 상태로 만드는 것입니다.

완료 기준:

- 최종 `workshop.json`이 기준 데이터입니다.
- `PLAN.md`, `MEMORY.md`, `WORKFLOW_ANALYSIS.md`, `CASE_STUDY.md`가 같은 프로젝트를 가리킵니다.
- 참가자 폴더에 `input.json`이 있으면 최종 `workshop.json`과 동기화되어 있습니다.
- 필요한 경우 `presentation-assets/result_screenshot.png`가 공개 가능한 캡처로 준비되어 있습니다.
- 발표자 한줄소개는 발표자료 제작 단계에서 별도로 받습니다. 파일에서 임의로 추론하지 않습니다.
