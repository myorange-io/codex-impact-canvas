# 캔버스 진행 흐름

이 흐름은 한국어로 진행합니다. 한 번에 하나의 질문만 묻고, 답변을 받은 뒤 1-2문장으로 요약합니다. 이 스킬은 신청서 내용을 다시 묻는 인터뷰가 아니라, 참여자가 준비해 온 현장 문제를 3시간 MVP 1개와 AI Agent Workflow 자산으로 압축하는 워크숍 코치입니다.

## 목차

- 질문 도구 사용
- 6단계 진행 흐름
- 체크리스트로 쓰는 기존 앵커
- 수렴 규칙
- 산출물 안내

## 질문 도구 사용

이 문서의 `질문`, `부족할 때만 묻는 질문`, `수렴 규칙` 문구는 사용자에게 그대로 채팅으로 보내는 문장이 아니라 `AskUserQuestion` 도구 호출의 재료입니다.

- 사용자에게 새 정보를 받아야 하면 항상 `AskUserQuestion`을 호출합니다.
- 한 번의 호출에는 질문 1개만 넣고, 답변을 받은 뒤 요약한 다음 다음 호출 여부를 판단합니다.
- 이미 확인된 내용은 다시 묻지 않습니다. 빈칸이 있을 때만 `AskUserQuestion`을 호출합니다.
- 자유 서술 답변이 필요한 단계에서는 선택지를 답변 방식으로 구성합니다. 예: `짧게 직접 입력`, `자료를 먼저 붙여넣기`, `가상 샘플로 시작`, `직접 입력`.
- 유형 선택, 결과물 선택, MVP 압축처럼 선택이 필요한 단계에서는 모든 선택지가 실제로 3시간 MVP가 될 수 있어야 합니다. 명백한 정답 하나와 들러리 선택지를 만들지 않습니다.
- 선택지 설명은 한 줄로 쓰고, 그 선택을 고르면 무엇이 확정되는지 밝힙니다.

기본 호출 형태:

```text
AskUserQuestion
question: "[아래 단계의 질문 문구를 상황에 맞게 한 문장으로 압축]"
multiSelect: false
options:
  - label: "[추천 또는 가장 자연스러운 답변 방식]"
    description: "[이 선택으로 확정되는 것]"
  - label: "[다른 실행 가능한 선택]"
    description: "[이 선택으로 확정되는 것]"
  - label: "직접 입력"
    description: "위 선택지와 다르게 직접 설명합니다."
```

## 6단계 진행 흐름

### 1. 기본 정보 확인

먼저 팀 시작 정보를 받습니다.

- 사회혁신가 이름
- 개발자 이름
- 팀 이름 또는 프로젝트 이름. 이미 정해져 있을 때만 받습니다.

프로젝트 이름이 없다면 문제 설명에서 짧은 임시 제목을 만들고, 임시 제목이라고 표시합니다.

질문:

```text
오늘 작업할 팀 정보를 먼저 확인하겠습니다. 사회혁신가 이름, 개발자 이름, 정해진 팀/프로젝트 이름을 알려주세요.
```

완료 기준:

- 사회혁신가와 개발자 이름이 기록됩니다.
- 팀명 또는 프로젝트명이 있으면 기록됩니다.
- 프로젝트명이 없으면 임시 제목을 만들 수 있습니다.

### 2. 업무 문제 확인

샘플 입력이나 기대 산출물을 바로 묻지 않습니다. 먼저 실제 업무 장면을 확인합니다.

질문:

```text
지금 반복해서 처리하느라 불편한 업무는 무엇이고, 누가 어떤 순간에 가장 막히나요?
```

완료 기준:

- 반복 처리 업무가 드러납니다.
- 불편을 겪는 사람이 직접 사용자, 검토자, 최종 수혜자 중 누구인지 구분됩니다.
- 문제가 발생하는 순간이 한 문장으로 정리됩니다.

부족할 때만 묻는 질문:

- "가장 자주 반복되는 업무 1개만 고르면 무엇인가요?"
- "이 문제 때문에 가장 시간을 쓰거나 곤란해지는 사람은 누구인가요?"
- "문제가 실제로 터지는 순간을 하나만 말하면 언제인가요?"

### 3. 자료 확인

업무 문제를 들은 뒤 실제 자료 상태를 확인합니다. 자료가 없으면 가상 샘플을 만들 수 있습니다.

질문:

```text
오늘 써볼 실제 자료가 있나요? 있다면 개인정보를 제거한 샘플로 쓸 수 있고, 없다면 가상 샘플로 시작해도 됩니다.
```

완료 기준:

- 실제 자료 유무가 정해집니다.
- 자료 형태가 파일, 표, 문서, 질문 묶음, 화면, 텍스트, 없음 중 하나로 기록됩니다.
- 개인정보 처리 방식이 익명화, 마스킹, 삭제, 더미 생성, 화면 시연 중 하나로 정해집니다.
- 자료가 없으면 스킬이 만들 가상 샘플의 조건이 정해집니다.

자료 유효성 확인 기준:

- 3시간 안에 열어보고 처리할 수 있는 크기입니다.
- 발표장에서 보여줘도 되는 수준으로 개인정보와 내부 원문이 제거되어 있습니다.
- 입력 -> 처리 -> 판단 -> 산출물 흐름을 보여줄 최소 정보가 있습니다.
- 원본을 공개할 수 없으면 `가상 샘플` 또는 `익명화 샘플`로 표시합니다.

### 4. 결과물 선택

`references/problem-archetypes.md`로 문제 유형을 가늠하고, `references/output-types.md`로 가능한 결과물 후보를 제안합니다. 이 단계에서 스킬이 후보를 먼저 만듭니다.

질문:

```text
이 문제는 `[문제 유형]`에 가깝습니다. 오늘 결과물은 아래 후보 중 무엇으로 보여주는 것이 가장 자연스러울까요?
```

완료 기준:

- 문제 유형이 문서/콘텐츠, 데이터/대시보드, 문의응대/매뉴얼, 분류/추천/리서치, 웹앱/시스템, MVP 미정형 중 하나로 기록됩니다.
- 결과물 형태가 자동화 도구, 웹앱, 온보딩 페이지, admin 화면, 대시보드, 문서/메시지 초안, 기타 중 하나로 기록됩니다.
- 결과물 후보는 모두 실행 가능해야 하며, 하나만 진짜 정답처럼 보이면 다시 구성합니다.

### 5. 3시간 MVP 좁히기

결과물 형태가 정해진 뒤 입력 1개, 출력 1개, 사람 검토 1개로 완결형 업무 흐름을 자릅니다.

질문:

```text
3시간 안에 보여줄 핵심 흐름 1개만 남기면, 어떤 입력을 넣어 어떤 결과를 보여주고 사람이 어디를 검토하면 될까요?
```

완료 기준:

- 핵심 기능 1개가 있습니다.
- 데모 입력과 데모 산출물이 정해집니다.
- 오늘 할 것과 하지 않을 것이 분리됩니다.
- AI 역할과 사람 검토 지점이 분리됩니다.
- 성공 기준과 실패 신호가 샘플 결과로 판단 가능합니다.

부족할 때만 묻는 질문:

- "AI가 맡을 일과 사람이 반드시 판단할 일을 나누면 어떻게 되나요?"
- "오늘 하지 않을 일 1-3개를 고르면 무엇인가요?"
- "샘플 입력 1개를 넣었을 때 어떤 결과가 나오면 성공이고, 어떤 결과가 나오면 실패인가요?"

### 6. 다음 단계 안내

문제정의가 끝나면 `PLAN.md`와 `workshop.json` 초안을 만들고, 구현이 끝난 뒤 후속 산출물로 이어진다는 점을 안내합니다.

질문:

```text
문제정의는 `PLAN.md`와 `workshop.json` 초안으로 저장하겠습니다. 구현 후에는 기록을 바탕으로 워크플로 분석과 공개 사례까지 만들면 될까요?
```

완료 기준:

- `PLAN.md` 초안의 용도가 팀 합의와 3시간 실행 계획임을 안내했습니다.
- `workshop.json`의 용도가 제출/아카이빙/산출물 재생성 기준 데이터임을 안내했습니다.
- 구현 중 `MEMORY.md`를 append-only로 기록할 것을 안내했습니다.
- 완료 후 `WORKFLOW_ANALYSIS.md`, `CASE_STUDY.md`, 최종 `workshop.json`을 만들 것을 안내했습니다.

## 체크리스트로 쓰는 기존 앵커

아래 항목은 순서대로 반복 질문하지 않습니다. 사용자가 준비해 온 설명에서 이미 확인된 내용은 건너뛰고, 빈칸만 짧게 묻습니다.

- 실제 업무 문제: 누가, 언제, 무엇 때문에 막히는가?
- 어려움을 겪는 사람: 직접 사용자, 검토자, 최종 수혜자가 구분되는가?
- 현재 업무 흐름: 입력 -> 처리 -> 판단 -> 산출물이 보이는가?
- 자료 상태: 실제 자료, 익명화 샘플, 가상 샘플 중 무엇을 쓸 것인가?
- 결과물 형태: 발표장에서 무엇을 눈으로 보여줄 것인가?
- 기대 변화: 시간 절감, 오류 감소, 누락 방지, 응답 속도 개선, 일관성 개선 중 관찰 가능한 변화가 있는가?
- 3시간 MVP: 입력 1개, 출력 1개, 사용자 행동 1개, 사람 검토 1개로 줄었는가?
- 성공 판단 기준: 샘플 결과를 보고 성공/실패를 판단할 수 있는가?
- 제출/아카이빙: 최종 결과물과 기록이 워크플로 라이브러리로 재사용 가능한가?

## 수렴 규칙

팀이 여러 아이디어를 나열하면 하나의 완결형 업무 흐름을 고르게 합니다.

```text
3시간 MVP에서는 하나만 남기겠습니다. 오늘은 `[입력]`을 넣어 `[결과물]`을 만들고 `[사람 검토]`에서 확인하는 흐름으로 두고, `[나머지]`는 제외 범위나 다음 단계로 기록해도 될까요?
```

시스템급 요청은 오늘 만들 화면, 문서, 표, 프롬프트, 워크플로 중 하나로 줄입니다.

```text
이건 완성된 시스템으로는 3시간을 넘습니다. 오늘은 `[사용자 행동 1개]`와 `[결과 화면/문서 1개]`가 보이는 데모로 자르고, 연동과 자동 실행은 다음 단계로 빼겠습니다.
```

답변이 계속 추상적이면 반복 업무 하나부터 고르게 합니다.

```text
아직 기능이 정해지지 않았다면, 가장 자주 반복되는 업무 하나만 먼저 고르겠습니다. 오늘 사람이 덜 해도 되는 일 1개는 무엇인가요?
```

자료가 없으면 가상 샘플을 제안합니다.

```text
실제 자료가 없어도 진행할 수 있습니다. 오늘은 `[업무 상황]`을 반영한 가상 샘플 1개로 입력과 결과물을 고정해도 될까요?
```

## 산출물 안내

산출물은 워크숍 단계별로 역할이 다릅니다.

- 문제정의 후: `PLAN.md`, `workshop.json` 초안
- 구현 중: `MEMORY.md` append, `workshop.json` 업데이트
- 완료 후: `WORKFLOW_ANALYSIS.md`, `CASE_STUDY.md`, 최종 `workshop.json`

제출/아카이빙 기준:

- `PLAN.md`에는 팀 합의, 자료 상태, 결과물 형태, 3시간 MVP, 제외 범위가 있습니다.
- `workshop.json`에는 질문 답변과 구현 결과가 표준 필드로 들어 있습니다.
- `MEMORY.md`에는 구현 중 결정과 변경이 누적되어 있습니다.
- `WORKFLOW_ANALYSIS.md`에는 재사용 가능한 AI Agent Workflow 필드가 채워져 있습니다.
- `CASE_STUDY.md`에는 공개 가능한 수준의 익명화 사례만 들어 있습니다.
