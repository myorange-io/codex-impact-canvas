# 산출물 일관성

워크숍 마크다운 산출물을 새로 만들거나 수정하기 전에 이 문서를 확인합니다. 파일들이 나중에 "AI 에이전트를 활용한 사회문제 해결 사례집" 또는 "사회혁신가를 위한 AI 에이전트 워크플로 라이브러리"로 합쳐질 수 있으므로 일관성이 중요합니다.

## 필수 파일

아래 파일은 항상 만들거나 유지합니다. 산출물 범위는 사용자에게 선택 질문으로 묻지 않고 워크숍 표준 흐름을 기본으로 적용합니다. 마크다운 파일과 구조화 데이터 파일을 구분합니다.

마크다운 산출물:

- `PLAN.md`
- `MEMORY.md`
- `WORKFLOW_ANALYSIS.md`
- `CASE_STUDY.md`

구조화 데이터:

- `workshop.json`

필수 파일명은 바꾸지 않습니다. `CASE_STUDY.md`와 `workshop.json`은 선택이 아니라 필수입니다.

## 표준 상태값

아래 값을 정확히 사용합니다.

- 알 수 없음: `미정`
- 해당 없음: `해당 없음`
- 검토 필요: `검토 필요`
- 공개 범위: `공개 가능`, `익명화 후 공개 가능`, `내부 공유만`, `공개 불가`
- 개인정보 수준: `공개 데이터`, `내부자료`, `익명화 필요`, `개인정보 포함`, `민감정보 포함`
- 자료 상태: `실제 자료`, `익명화 샘플`, `가상 샘플`, `자료 없음`, `미정`
- 데이터 형태: `정형`, `반정형`, `비정형`, `혼합`, `미정`
- 결과물 형태: `자동화 도구`, `웹앱`, `온보딩 페이지`, `admin 화면`, `대시보드`, `문서/메시지 초안`, `기타`, `미정`
- 연결 수준: `수동 업로드`, `CLI`, `MCP`, `API`, `브라우저`, `SaaS`, `미정`
- MCP/CLI 연결 여부: `MCP 사용`, `CLI 사용`, `MCP+CLI 사용`, `사용 안 함`, `미정`
- 사람 검토: `최종 검토`, `중간 검토`, `예외 검토`, `검토 없음`, `미정`
- 재사용 수준: `그대로 재사용 가능`, `템플릿 재사용 가능`, `분야 특화`, `일회성`, `미정`

## 산출물 순서

각 파일의 섹션 순서는 고정합니다. 정보가 없으면 제목을 유지하고 `미정`이라고 씁니다.

짧은 bullet list와 표를 우선 사용합니다. 단, `CASE_STUDY.md`에서 공개 설명이 필요한 섹션은 자연스러운 문단을 허용합니다.

워크숍 단계별 산출물 역할은 아래처럼 유지합니다.

- 문제정의 후: `PLAN.md`, `workshop.json` 초안
- `PLAN.md` 작성 직후: 작성된 `PLAN.md` 전체 내용을 채팅창에 보여준 뒤, 스킬이 직접 구현하지 않고 사용자가 `PLAN.md` 기준으로 구현
- 사용자가 구현 중: 새 세션 구현자도 `MEMORY.md` append-only 갱신, `workshop.json` 업데이트 대상 기록
- 사용자가 구현 완료 후: `MEMORY.md`, `WORKFLOW_ANALYSIS.md`, `CASE_STUDY.md`, 최종 `workshop.json`
- 발표자료 제작 전: 최종 `workshop.json` 기준 참가자 폴더, 사용자 제공 발표자 한줄소개, 선택적 `presentation-assets/result_screenshot.png`

## YAML 메타데이터

`PLAN.md`, `WORKFLOW_ANALYSIS.md`, `CASE_STUDY.md`에는 아래 YAML 메타데이터를 넣습니다.

```yaml
---
team_name: ""
project_name: ""
created_date: "YYYY-MM-DD"
event: "Codex Impact Workshop"
version: "1"
---
```

세 파일에서 `team_name`, `project_name`, `created_date`, `event` 값은 동일해야 합니다.

## 안정적인 사례 ID

라이브러리화를 위해 `WORKFLOW_ANALYSIS.md`와 `CASE_STUDY.md`에 같은 `case_id`를 넣습니다.

형식:

```text
impact-agent-[YYYYMMDD]-[short-slug]
```

slug는 소문자 ASCII를 사용하고, 공백은 하이픈으로 바꿉니다.

프로젝트 이름이 한국어라면 `project.slug`에 짧은 영문 식별자를 따로 넣는 것을 권장합니다. 스크립트는 slug가 없을 때 `case-xxxxxxxx` 형태의 안정적인 대체 slug를 만듭니다.

## 품질 점검

마무리 전 아래를 확인합니다.

- 네 개의 필수 마크다운 파일이 모두 존재합니다.
- `workshop.json`이 존재하고, 마크다운 산출물을 재생성할 수 있는 표준 필드를 포함합니다.
- `PLAN.md`에는 선택된 3시간 MVP 기능이 정확히 1개만 있습니다.
- `PLAN.md`에는 자료 상태, 개인정보 처리, 선택한 결과물 형태, 해결할 좁은 문제, 포함할 업무 단계, 사람 검토 지점, 실패 신호, 3시간 안/밖 판단이 있습니다.
- `PLAN.md`에는 새 세션 구현자도 `MEMORY.md`를 append-only로 갱신하라는 구현자 지침이 있습니다.
- `PLAN.md` 작성 완료 후에는 파일 내용을 채팅창에 보여주고, 이제 구현을 시작하면 되며 구현 완료 후 발표자료 제작을 요청하면 된다고 안내합니다.
- 익명화되지 않은 샘플은 원본 그대로 쓰지 않고 오렌지필터 설치 후 익명화하도록 안내되어 있습니다.
- `PLAN.md`에는 핵심 기능이 빨리 끝났을 때 이어서 할 작업 후보 2개가 있고, 각 후보에는 시작 조건, 구현 내용, 데모 산출물, 예상 추가 시간, 검수 기준이 있습니다.
- `WORKFLOW_ANALYSIS.md`에는 데이터 특성, 정형/비정형 여부, 데이터 수, 개인정보 포함 여부, MCP/CLI 연결 여부, 입력 데이터 출처, 산출물 형식, 반복 빈도, 기존 대비 소요 시간 변화, 주요 검수 포인트, 권한/접근 이슈, 재사용 가능성, 실패 시 사람이 이어받는 방식, 결과물의 공유 가능 범위가 있습니다.
- `CASE_STUDY.md`에는 원본 개인정보, 비밀값, 내부 문서 원문, 접근 방법, 토큰, 익명화되지 않은 원본 데이터가 없습니다.
- 발표자료 제작에 넘길 기준 데이터는 최종 `workshop.json`입니다. `input.json`이 참가자 폴더에 있으면 최종 `workshop.json`과 충돌하지 않는지 확인합니다.
- 결과물 캡처를 쓰는 경우 `presentation-assets/result_screenshot.png`에 공개 가능한 익명화 이미지가 있습니다.
- 알 수 없는 필드는 `미정`을 사용하고, 필수 제목을 조용히 삭제하지 않습니다.
- 추가 아이디어는 MVP에 섞지 않고 제외 범위나 다음 단계에 기록합니다.

## 권장 자동화

가능하면 답변을 JSON으로 정규화한 뒤 아래 명령을 실행합니다.

```bash
python3 scripts/write_workshop_outputs.py --input workshop.json --output-dir . --phase all
```

이 스크립트는 모든 팀의 산출물 제목과 값을 같은 구조로 유지하고, 출력 디렉터리에 표준화한 `workshop.json`도 함께 저장합니다.

문제정의 직후에는 최종 분석 문서를 미리 만들지 않도록 `--phase plan`을 사용합니다. 이 단계에서 스킬은 생성된 `PLAN.md`를 다시 읽어 채팅창에 보여준 뒤, 직접 구현하지 않고 사용자에게 `PLAN.md` 기준 구현을 넘깁니다. 구현 완료 후 분석과 사례 정리를 만들 때는 `--phase final`을 사용합니다.
