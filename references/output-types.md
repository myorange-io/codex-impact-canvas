# 결과물 형태 선택 기준

업무 문제를 들은 뒤 가능한 결과물 후보를 제안할 때 사용합니다. 결과물 형태는 문제 유형과 다릅니다. 문제 유형은 질문을 줄이기 위한 분류이고, 결과물 형태는 발표장에서 실제로 보여줄 구현물입니다.

## 공통 원칙

- 결과물 후보는 2-4개만 제안합니다.
- 모든 후보는 3시간 안에 최소 데모가 가능해야 합니다.
- 명백한 정답 하나와 들러리 선택지를 만들지 않습니다.
- 각 후보 설명에는 입력, 출력, 사람 검토 지점을 한 줄로 적습니다.
- 로그인, 결제, 권한, 자동 발송, 외부 API 연동, 실시간 DB 저장은 기본적으로 다음 단계로 둡니다.

## 결과물 형태

### 1. 자동화 도구

적합한 상황:

- 이 프로젝트의 반복 자료, 표, 기록, 문의를 처리합니다.
- 사용자는 결과 파일이나 표만 받아도 업무가 줄어듭니다.

프로젝트 기반 MVP 예시 구성:

- `[이 프로젝트의 자료 묶음] -> [담당자가 검토할 표나 문서]`
- `[이 프로젝트의 원본 텍스트/표] -> [사람이 확인할 요약, 분류, 초안]`
- 예시는 사용자가 말한 실제 자료와 결과물 이름으로 다시 작성합니다.

사람 검토 지점:

- 분류 기준, 누락 항목, 최종 발송/확정 전 검토

### 2. 웹앱

적합한 상황:

- 사용자가 입력하고 바로 결과를 확인해야 합니다.
- 화면이 있어야 발표장에서 흐름이 이해됩니다.

프로젝트 기반 MVP 예시 구성:

- `[이 프로젝트의 입력 화면] -> [결과 확인 화면]`
- `[이 프로젝트의 사용자 행동 1개] -> [사람이 수정/확정할 결과 화면]`
- 예시는 사용자가 말한 실제 입력값, 화면, 결과로 다시 작성합니다.

사람 검토 지점:

- 결과 화면에서 수정, 확정, 보류 중 하나를 고르는 지점

### 3. 온보딩 페이지

적합한 상황:

- 이 프로젝트의 대상자가 절차를 따라야 합니다.
- 자료 안내와 행동 유도가 중요합니다.

프로젝트 기반 MVP 예시 구성:

- `[이 프로젝트의 절차/자료] -> [대상자가 따라야 할 안내 페이지]`
- `[이 프로젝트의 참여자 행동] -> [개인 또는 상황별 체크리스트]`
- 예시는 사용자가 말한 실제 대상자, 절차, 안내 목적에 맞춰 다시 작성합니다.

사람 검토 지점:

- 안내 문구의 정확성, 민감한 표현, 조직 정책 반영 여부

### 4. admin 화면

적합한 상황:

- 담당자가 여러 항목을 검토, 승인, 배정, 라우팅해야 합니다.
- 최종 판단은 사람이 해야 합니다.

프로젝트 기반 MVP 예시 구성:

- `[이 프로젝트의 검토 대상 묶음] -> [담당자가 판단할 플래그/메모 화면]`
- `[이 프로젝트의 여러 항목] -> [배정, 보류, 우선순위 등 사람 판단 화면]`
- 예시는 사용자가 말한 실제 검토 대상과 판단 기준으로 다시 작성합니다.

사람 검토 지점:

- 승인/보류/배정/최종 판단 버튼 또는 체크리스트

### 5. 대시보드

적합한 상황:

- 반복적으로 지표를 확인하거나 비교해야 합니다.
- 표나 그래프가 의사결정을 돕습니다.

프로젝트 기반 MVP 예시 구성:

- `[이 프로젝트의 반복 확인 데이터] -> [핵심 지표나 비교표]`
- `[이 프로젝트의 표/목록] -> [현장에서 해석할 요약과 근거]`
- 예시는 사용자가 말한 실제 데이터, 지표, 비교 기준으로 다시 작성합니다.

사람 검토 지점:

- 계산 기준, 필터 기준, 해석 문구 확인

### 6. 문서/메시지 초안

적합한 상황:

- 최종 산출물이 이 프로젝트에서 실제로 쓰는 문서나 메시지 초안입니다.
- 자동 발송보다 초안 품질과 검수가 중요합니다.

프로젝트 기반 MVP 예시 구성:

- `[이 프로젝트의 원본 문서/기록] -> [최종 발송 전 사람이 다듬을 초안]`
- `[이 프로젝트의 반복 작성 자료] -> [사실관계와 표현을 검토할 문서/메시지]`
- 예시는 사용자가 말한 실제 원본과 최종 산출물 이름으로 다시 작성합니다.

사람 검토 지점:

- 최종 표현, 사실관계, 발송 가능 여부
