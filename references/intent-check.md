# 의도 검증

최종 산출물을 확정하기 전에 합의한 MVP 의도와 실제 파일 내용을 비교합니다. 문서가 구조적으로 맞아도 의도와 어긋나면 수정 질문을 한 번 더 합니다.

## 검증 원칙

- 문서는 증거가 아니라 확인 대상입니다. 합의한 의도가 파일에 실제로 반영됐는지 확인합니다.
- 문제, 입력, 출력, 사람 검토, 실패 대응, 데이터 공유 제한은 서로 연결되어야 합니다.
- 새 기능이나 큰 시스템 욕구가 산출물에 몰래 섞이면 제외 범위나 다음 단계로 옮깁니다.
- 공개 사례에는 개인정보, 내부 원문, 접근권한 세부 정보, 비밀값, 토큰을 넣지 않습니다.

## 의도 요약

산출물을 쓰기 전에 아래 6개를 짧게 고정합니다.

```markdown
## 합의한 MVP 의도
- 문제 유형:
- 샘플 입력 1개:
- 기대 출력 1개:
- 핵심 기능 1개:
- 사람이 검토할 지점:
- 실패 시 이어받는 방식:
```

## 파일별 확인

### PLAN.md

- `## 3시간 MVP`의 핵심 기능이 합의한 핵심 기능 1개와 같은가?
- 데모 입력과 데모 산출물이 합의한 입력/출력과 같은가?
- 성공 기준이 샘플 결과로 판단 가능한가?
- 제외 범위에 자동 연동, 배포, 일괄 처리, 대형 시스템 기능이 빠져 있는가?
- 빠르게 끝났을 때 할 후보 2개가 핵심 기능을 흔들지 않는가?

### WORKFLOW_ANALYSIS.md

- 기존 업무 흐름에 입력 -> 처리 -> 판단 -> 산출물이 보이는가?
- 자동화 대상이 업무 전체가 아니라 오늘 줄일 반복 행동 1개인가?
- AI 에이전트 입력 조건과 산출물 조건이 샘플 기준으로 구체적인가?
- 사람 검토 지점, 실패 시 이어받는 방식, 예상 오류 유형이 비어 있지 않은가?
- 개인정보 처리와 결과물 공유 가능 범위가 데이터 제한과 맞는가?

### CASE_STUDY.md

- 문제 요약이 공개 가능한 수준으로 익명화되어 있는가?
- AI 에이전트 워크플로가 실제 MVP 흐름을 과장하지 않는가?
- 사용 도구, MVP 결과물, 성공 기준이 `PLAN.md`와 충돌하지 않는가?
- 익명화와 공개 제외 섹션에 공개하지 않을 정보가 명시되어 있는가?

## 수정 질문

의도와 산출물이 어긋나면 아래 중 하나만 짧게 묻습니다.

- "지금 산출물에는 `[내용]`까지 들어갔는데, 3시간 MVP에서는 `[핵심 기능]`만 남기고 나머지는 다음 단계로 빼도 될까요?"
- "성공 기준이 아직 효과 수준입니다. 샘플 입력 1개를 넣었을 때 성공/실패를 가르는 결과 기준을 하나로 말하면 무엇인가요?"
- "사람 검토 지점이 비어 있습니다. AI 결과를 발표 전에 사람이 꼭 확인해야 하는 부분은 어디인가요?"
- "데이터 공유 제한이 산출물에 반영되지 않았습니다. 공개 사례에서 반드시 숨길 정보는 무엇인가요?"

## 완료 기준

- 핵심 MVP가 모든 파일에서 같은 의미로 쓰입니다.
- 성공 기준, 사람 검토, 실패 대응, 데이터 공유 제한이 빠지지 않습니다.
- 다음 단계 아이디어가 MVP 본문에 섞이지 않습니다.
- `CASE_STUDY.md`가 공개 가능한 수준으로 익명화되어 있습니다.
